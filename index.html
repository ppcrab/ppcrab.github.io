突破口—-strtoul

strtoul()函数我之前就没用过~，但是在这个程序中使用的次数十分多，起到了十分关键的作用。经过搜索之后发现这个函数的作用是把字符串类型的一串字符转换成相应的int类型的数字（第三个参数对应着转换的进制），再详细的记录我觉得没有太大的必要，用的时候自己去搜就好。
这个函数使用的时候需要注意的一点就是返回值，返回值为无符号long型数值，如果是负数会返回一个大正数。也就是说，这个这个东西虽然是转换字符串类型为int类型的，但是并不是对于任何的数值都是这么的好用，对于’-100’这种数值就无能为力了，而且还有返回数值emmm，虽然十分奇怪，真搞不懂为什么这种函数还会存在于C语言的库函数之中，明摆着是用来搞事情的吗。

fs段寄存器

在80X86中，段寄存器和与其对应存放偏移地址的寄存器之间有一种默认组合关系。
fs和gs都是80386起增加的两个辅助段寄存器,在这之前只有一个辅助段寄存器ES,增加这两个寄存器是为了减轻ES寄存器的负担,并能更好地配合适用于通用寄存器组的基址和变址寄存器.
这两个是通用的段寄存器，语法上同其它的段寄存器一样，不能直接用立即数给它赋值。

附加段寄存器

包括es.fs.gs
作用：
附加段寄存器ES，主要是用在字符串的操作上，由于在用一些串操作指令的时候，默认的目的串的段地址在ES中存放，所以必须设置Es。
在处理数据的时候,往往需要用到两个数据段,特别是在字符串的处理方面,使用两个数据段简便了许多的操作.
例如:字符串的匹配,查找,复制等等,设置了DS:SI,和ES:DI,很多的串操作指令就派上用场了,如REP REPNE CMPS等等…

核心转储文件

核心转储文件这个名词正式的进入我的脑海。。是在我第二次看《程序员的自我修养》的时候，因为我第一次看这本书的时候，是一知半解的，很迷糊的感觉~，第二次就好多了，看的比较清楚了，能够比较清晰的思考其中的问题了（原谅我的菜啊~~~）

维基定义：核心文件（core file），也称核心转储（core dump），是操作系统在进程收到某些信号而终止运行时，将此时进程地址空间的内容以及有关进程状态的其他信息写出的一个磁盘文件。这种信息往往用于调试。linux下的是core文件（core dump），windows下的是Memory.dmp
使用：核心文件通常在系统收到特定的信号时由操作系统生成。信号可以由程序执行过程中的异常触发，也可以由外部程序发送。动作的结果一般是生成一个某个进程的内存转储的文件，文件包含了此进程当前的运行堆栈信息。有时程序并未经过彻底测试，这使得它在执行的时候一不小心就会遭到破坏。这可能会导致核心转储（core dump）。现在的UNIX系统极少会面临这样的问题。即使遇到，程序员可以通过核心映像调试程序来找到错误原因。

其实这个东西引用十分广泛，有的时候当我们在运行某些软件，会遇到程序崩溃的问题。出现这种情况的原因有的时候是因为一些不知名错误，莫名奇妙的。有的时候就只是卡死了~。
windows10在面临这种情况的时候，会表示记录下程序出现的问题，然后发到（额，我忘了具体的名字了）一个地方去。我现在在想，应该就是这个东西了。（可以确定的是，windows蓝屏的时候一定有这个文件的存在，大佬们可以使用这个文件中的信息来进行错误诊断）
至于我们经常使用的linux，就会更经常见到这个东西了。
首先，gdb跟踪调试是linux环境下调试常用的方法，但在解决一些问题，如频繁创建线程，打印信息很多时，gdb调试会显得效率很低。这个时候，可以先生成内核转储文件，再针对生成的转储文件进行调试，从而大幅度提高调试效率。具体的命令可以很轻松的Google到，在此不再重复，因为本人现在还没有很深入的使用这个东西。

gcc -S 会生成一个.s文件，然后直接进行cat这个这个.s文件就可以看到这个程序的汇编代码了。
gcc -g 可以产生调试信息，在linux中是core文件
gcc -E 只把源文件进行预处理之后的结果输出来。不做编译，汇编，连接的动作。
gcc -c表示只编译(compile)源文件但不链接，会把.c或.cc的c源程序编译成目标文件，一般是.o文件。
gcc -o用于指定输出(out)文件名。不用-o的话，一般会在当前文件夹下生成默认的a.out文件作为可执行程序。
gcc -v把整个编译过程中的输出信息都给打印出来
gcc -Wall 显示编译中的所有的警告
